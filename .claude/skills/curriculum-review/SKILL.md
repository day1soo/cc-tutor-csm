# 커리큘럼-리뷰

> 핵심 질문: **"이 워크숍을 듣고 나서, 학습자가 실제로 자기 업무에 적용해보려고 할까?"**
> 상세 스펙: `requirements/curriculum-review-spec.md`
> 판단 기준: `references/review-rubric.md`

---

## 실행 방식

- `/커리큘럼-리뷰` → 전체 모듈 리뷰 (드라이런 **포함**, 기본)
- `/커리큘럼-리뷰 M2` → 특정 모듈만 리뷰 (드라이런 포함)
- `/커리큘럼-리뷰 빠르게` 또는 `/커리큘럼-리뷰 --no-dryrun` → 정적 분석만
- `/커리큘럼-리뷰 M3 빠르게` → 특정 모듈 + 정적만

**왜 드라이런이 기본인가:** 텍스트 정적 분석만으로는 "성공 기준 숫자 오류(5개 → 실제 6개)", "프롬프트 길이 체감(17줄)", "MCP auth 플로우 단계 수" 같은 **실제로 돌려봐야 보이는 이슈**를 놓친다. 드라이런은 이 구멍을 메운다.

---

## 실행 단계

### 1단계: 파일 수집

아래 파일들을 순서대로 읽는다:

1. **판단 기준**: `references/review-rubric.md` (안티패턴, 루브릭, 심각도 기준)
2. **리뷰 대상**: `.claude/skills/*/references/module-*.md` (인자로 모듈 지정 시 해당 모듈만)
3. **튜터 엔진**: `.claude/skills/*/SKILL.md` (cc-tutor-* 스킬의 메인 정의)
4. **프로젝트 규칙**: `CLAUDE.md`
5. **샘플 데이터**: `samples/` 폴더 내 모든 파일 (파일명 + 첫 5행)
6. **기존 피드백**: `FEEDBACK.md` (있으면 읽고, 없으면 건너뜀)

### 2단계: 중복 체크

FEEDBACK.md에 이미 지적된 항목을 파악한다.
- 모듈 번호 + 실습 번호 + 문제 키워드로 매칭
- 매칭된 항목은 리뷰 결과에서 `⏭️ FEEDBACK.md에서 기지적` 태그 후 상세 리뷰 건너뜀

### 2.5단계: 드라이런 (기본 ON, `빠르게` 인자 시 건너뜀)

각 모듈의 핵심 실습을 **CSM 초보 관점에서 실제로 돌려본다**. 결과를 `DRYRUN-NOTES.md`에 기록.

**드라이런 대상 자동 판별:**

| 실습 유형 | 검증 방법 |
|----------|----------|
| `samples/*.csv` 참조 | Read 실행 → 파일 존재 + 컬럼·행 수 확인 → 성공 기준 숫자와 대조 |
| `ls`·폴더 탐색 실습 | Bash `ls` 또는 Glob 실행 → 파일 개수·이름 확인 |
| 키워드 검색·필터링 | Grep 실제 실행 → 결과가 나오는지, 얼마나 나오는지 |
| MCP 실습 | ToolSearch로 스키마 확인 → auth 플로우 단계 수 검증 (자동 호출 X) |
| 프롬프트 길이 체감 | 코드 블록 **내부 줄 수** 카운트 → 3줄 초과 건 플래그 |

**🚨 안전 규칙 (반드시 준수):**
- **읽기·검색·ToolSearch만 허용**. 쓰기·외부 메시지 발송·실제 API 호출 금지
- MCP는 **스키마 확인까지만** — auth 실제 실행 X, 메시지 전송 X
- 학습자 데이터(my-data/)는 건드리지 않는다
- 수정 권한 있어도 **드라이런 중엔 절대 파일 수정 금지**

**DRYRUN-NOTES.md 구조:**
```markdown
# 드라이런 노트 — [날짜]

## M1 실습 결과
### 실습 N — [결과 / 발견]

## Slack MCP / Google Drive MCP 등 확인 결과
- 사용 가능 도구:
- auth 플로우:
- CSM 실습 연결성:

## 종합 발견사항 (리뷰에 반영)
- 🟣 [드라이런 전용 발견]
```

**리뷰 반영:** DRYRUN-NOTES.md의 발견사항은 3단계 리뷰 결과에 **🟣 드라이런 발견** 배지로 합류. 정적 분석 발견(🔴🟡🟢)과 색상으로 구분.

### 3단계: 리뷰 실행

`references/review-rubric.md`의 기준에 따라 3축으로 리뷰한다:

**축 1: 초보자 눈높이**
- 미설명 전문 용어 (glossary.md에 있지만 모듈에서 설명 없이 쓰인 것)
- 모호한 지시 ("이 폴더", "아래를" 등 참조가 불분명)
- 복붙 드릴 여부 (모듈 위치에 따른 점진적 자유도 기준 적용)

**축 2: 실습 동작**
- 실습 프롬프트가 참조하는 파일이 samples/에 존재하는지
- CSV 컬럼명이 프롬프트의 지시와 일치하는지
- 성공 기준이 실제 데이터로 달성 가능한지

**축 3: 학습 흐름**
- 이전 모듈에서 배운 개념이 다음 모듈에서 활용되는지
- CLAUDE.md에 정의한 규칙이 실습에서 실제로 반영되는지
- 난이도 점프가 갑작스럽지 않은지

### 4단계: 안티패턴 + 동기 부여 체크

**3대 안티패턴** (review-rubric.md 상세 기준 참조):
1. 복붙 드릴 — 학습자 질문 설계 기회 없음
2. 실무-실습 갭 — 본인 맥락 변환 기회 없음
3. 트리거 부재 — "실무 상황 + 도구" 연결 없음

**동기 부여 3기준**:
1. Before/After 체감이 눈에 보이는가
2. 내 업무와 직결되는가
3. 점진적 복잡도가 자연스러운가

### 5단계: 심각도 분류

각 발견 항목에 심각도를 부여한다:
- **🔴 치명적**: 안 고치면 학습자가 이탈 (실습 막힘, 후반부 완전 복붙, 미설명 전문용어)
- **🟡 개선 필요**: 안 고쳐도 되지만 교육 효과 떨어짐 (트리거 부재, 맥락 변환 없음)
- **🟢 제안**: 더 좋아질 수 있는 포인트 (Before/After 강화, 표현 개선)

### 6단계: HTML 생성

`references/review-template.html`을 기반으로 `REVIEW.html` 파일을 생성한다.

**HTML 구조:**
- 상단 대시보드: 심각도별 집계 카드 (🔴🟡🟢) + 🟣 드라이런 발견 카드 + "가장 먼저 고칠 것" Top 3
- 모듈별 섹션: 접기/펼치기 가능
- 각 항목: 심각도 컬러 카드 (문제 → 왜 문제인가 → 수정 방향)
- 🟣 드라이런 발견 항목은 **보라색 배지**로 구분 (정적 분석과 시각적 분리)
- 하단: 리뷰 메타 정보 (일시, 대상 모듈, FEEDBACK.md 중복 제외 수, 드라이런 여부)

### 7단계: 안내

리뷰 완료 후 아래 메시지를 출력한다:

```
✅ 리뷰 완료! REVIEW.html이 생성되었습니다.
브라우저에서 열어보세요: REVIEW.html

📊 요약:
- 🔴 치명적: N건
- 🟡 개선 필요: N건
- 🟢 제안: N건
- 🟣 드라이런 발견: N건 (드라이런 수행 시)
- ⏭️ FEEDBACK.md 기지적: N건 (건너뜀)

드라이런 상세 로그: DRYRUN-NOTES.md (드라이런 수행 시)
```

## 톤/깊이 레퍼런스

**FEEDBACK.md에 연수님이 직접 작성한 피드백이 이 리뷰의 품질 기준선(gold standard)이다.**

리뷰 항목을 작성할 때:
- FEEDBACK.md의 깊이와 톤을 참조한다
- 예: "실습 4가 메타 질문에 그침 → 규칙이 실제로 적용되는 장면을 학습자가 못 본다" — 이 수준의 구체성을 목표로 한다
- 단순 "이 부분이 모호합니다"가 아니라, **왜 모호한지, 학습자가 어떤 경험을 하게 되는지**까지 서술한다

---

## 주의사항

- 리뷰만 한다. 모듈 파일을 직접 수정하지 않는다. **드라이런 중에도 수정 금지**.
- FEEDBACK.md는 읽기만 한다. 쓰지 않는다.
- HTML 파일은 매번 덮어쓴다 (이전 리뷰 결과는 유지하지 않음).
- `DRYRUN-NOTES.md`는 드라이런 시 생성·덮어쓰기. 정적 모드(`빠르게`)에선 건드리지 않음.
- 모듈이 없는 번호(M5, M6 등)를 지정하면 "해당 모듈 파일이 없습니다" 안내.
- **드라이런 안전 규칙** (재강조): 읽기·검색·ToolSearch만. 외부 메시지 발송·API write·파일 수정 절대 금지.
