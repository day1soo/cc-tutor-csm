# 커리큘럼 리뷰 루브릭

> 이 문서는 `/커리큘럼-리뷰` 스킬의 판단 기준이다.
> SKILL.md에서 참조하여 사용한다.

---

## 3대 안티패턴

### 1. 복붙 드릴

학습자가 질문을 설계하지 않고, 주어진 프롬프트를 복사-붙여넣기만 하는 구조.

**판별 기준:**

| 체크 | 질문 |
|------|------|
| 질문 설계 | 학습자가 **본인 질문을 만들** 기회가 있는가? (예: "궁금한 걸 골라서 물어보세요") |
| 결과 예측 | 실행 전에 **"이게 나올 것 같나요?"** 예상해볼 기회가 있는가? |
| 프롬프트 구조 | "아래를 그대로 입력하세요"로만 구성되어 있는가? |

**점진적 자유도 (모듈 위치별 기준):**

| 모듈 위치 | 복붙 비중 | 판단 |
|----------|----------|------|
| M1 (초반) | 70~80% OK | 도구에 익숙해지는 단계. 복붙 위주 자연스러움. |
| M2 (중반) | 50% 이하 권장 | 감각이 생긴 단계. 자유 질문/응용이 섞여야 함. |
| M3~M4 (후반) | 30% 이하 권장 | 자유 질문/응용이 주축. 복붙만이면 **🔴 치명적**. |

**좋은 예 vs 나쁜 예:**

| 나쁜 예 (복붙 드릴) | 좋은 예 (학습자 주도) |
|---|---|
| "아래 프롬프트를 입력하세요: `기업리스트에서 ongoing만 뽑아줘`" | "기업리스트에서 **본인이 궁금한 조건**으로 필터링해보세요. 예: ongoing만, High만, 이번 달 종료 예정..." |
| 프롬프트 → 결과 확인 → 다음 프롬프트 | 프롬프트 예시 제공 → **"이걸 변형해서 해봐도 됨"** 유도 → 결과 비교 |

---

### 2. 실무-실습 갭

샘플 데이터로만 실습하고, 본인 업무 맥락으로 변환할 기회가 없는 구조.

**판별 기준:**

| 체크 | 질문 |
|------|------|
| 데이터 경로 | `samples/` 고정 경로로만 실습하는가? |
| 변환 유도 | "내 파일로 해봐도 됨" 또는 `my-data/` 안내가 있는가? |
| 산출물 연결 | 실습 결과가 학습자의 **실제 업무 산출물**(리포트, 메일 등)과 닮았는가? |
| 상황 연결 | "이건 매월 리포트 만들 때 쓰면 돼" 같은 실무 상황 연결이 있는가? |

**좋은 예 vs 나쁜 예:**

| 나쁜 예 (갭 큼) | 좋은 예 (갭 작음) |
|---|---|
| `samples/enterprise-list-sample.csv`만 사용 | "샘플 대신 **본인 CSV**가 있으면 `my-data/`에 넣어서 해봐도 됩니다" |
| 분석 결과를 화면에서 보고 끝 | 결과를 **파일로 저장** → "이 파일을 내일 리포트에 바로 쓸 수 있어요" |

---

### 3. 트리거 부재

기능은 가르치지만, "실무에서 언제 이걸 떠올려야 하는지" 연결이 없는 구조.
→ 기능은 배우지만 돌아가면 **생각조차 안 남**.

**판별 기준 — "실무 상황 + 도구"가 한 문장에 연결되어 있는가?**

| 트리거 있음 ✅ | 트리거 없음 ❌ |
|---|---|
| "**매월 리포트 만들 때** 이 스킬을 쓰세요" | "Write 도구로 파일을 만들 수 있습니다" |
| "**파트장님 메시지가 오면** 이 스킬을 실행" | "SKILL.md는 이런 구조입니다" |
| "**기업 종료일이 다가오면** 이걸로 확인" | "Grep은 파일 내용을 검색하는 도구입니다" |

**체크 위치:**

| 섹션 | 체크 |
|------|------|
| WHY | "이 기능을 왜 배우는지" 실무 상황이 명시적인가? |
| EXECUTE | 실습 도입부에 "이건 실무에서 이럴 때 쓰는 거예요" 맥락이 있는가? |
| LEADER-TIP | "팀에서 언제 쓰면 되는지" 구체적 트리거가 있는가? |
| NEXT-STEP | 다음 모듈로 넘어가는 연결이 실무 흐름 기반인가? |

## 좋은 실습의 4대 기준

| 기준 | 판별 질문 | 충족 시 | 미충족 시 |
|------|----------|--------|---------|
| **질문 설계** | 학습자가 스스로 질문을 만드는 구조인가? | 🟢 이상 | 복붙 드릴 안티패턴 체크 |
| **맥락 변환** | 본인 데이터/상황으로 바꿀 여지가 있는가? | 🟢 이상 | 실무-실습 갭 안티패턴 체크 |
| **실무 트리거** | "이 상황이 오면 이걸 써"가 명시적인가? | 🟢 이상 | 트리거 부재 안티패턴 체크 |
| **Before/After** | 시간 절약이 구체적 수치로 보이는가? | 🟢 제안 수준 | 🟡 개선 필요 |

## 심각도 분류 체계

| 등급 | 의미 | 판단 기준 | 대표 예시 |
|------|------|----------|----------|
| **🔴 치명적** | 안 고치면 학습자 이탈 | 실습이 물리적으로 안 돌아감 / 후반부 완전 복붙 드릴 / 설명 없이 전문용어 / 참조 파일 없음 | `samples/없는파일.csv` 참조, M4에서 복붙만, "MCP"를 설명 없이 사용 |
| **🟡 개선 필요** | 교육 효과가 떨어짐 | 트리거 부재 / 맥락 변환 기회 없음 / 모듈 간 연결 약함 / Before/After 체감 약함 | WHY에 실무 상황 없음, samples/ 고정, M1 내용을 M2에서 안 씀 |
| **🟢 제안** | 더 좋아질 수 있음 | Before/After 수치화 가능 / 비유 보강 가능 / 성공 기준 구체화 가능 | 시간 절약 수치 추가, glossary 비유 활용, "→ ~~가 보이면 성공" 추가 |
| **🟣 드라이런 발견** | 실제로 돌려야 보이는 것 | 텍스트만으론 못 잡는 실제 수치·동작 불일치 | 성공 기준 "5개"인데 실제 6개, 프롬프트 17줄 체감, MCP auth 4단계 플로우 |

**🟣 심각도 재분류 규칙:** 드라이런 발견 항목은 **본질적으로 🔴/🟡/🟢 중 하나에 해당**한다. HTML 표시는 🟣 배지로 하되, 심각도 판단은 위 기준 그대로 적용. 예: "파일 경로 오류로 실습 막힘" = 🟣 + 🔴. "프롬프트 길이 체감 불편" = 🟣 + 🟡.

## 동기 부여 판단 기준

| 기준 | 체크 질문 | 좋은 신호 | 나쁜 신호 |
|------|----------|----------|----------|
| **Before/After 체감** | "수작업 N분 → 자동 N초" 같은 비교가 있는가? | WHY에 구체적 시간/노력 비교표 | "편리합니다" 같은 추상적 설명만 |
| **내 업무 연결성** | 실습이 학습자의 실제 반복 업무와 닮았는가? | 기업리스트, 리포트 등 실제 CSM 업무 데이터 | 일반적인 예제 (Todo 앱, Hello World) |
| **점진적 복잡도** | 쉬운 것 → 어려운 것 순서가 자연스러운가? | M1(파일 읽기) → M2(분석) → M3(MCP) → M4(스킬) | 갑자기 난이도 점프 (M1에서 바로 MCP) |

## 리뷰 항목 작성 포맷

각 발견 항목은 반드시 아래 3단계로 작성한다:

```
### [심각도] [모듈번호] [실습/섹션] 제목 한 줄

- **문제**: 무엇이 문제인지 (구체적 위치 포함)
- **왜 문제인가**: 초보 학습자 관점에서 왜 이게 걸리는지 / 어떤 경험을 하게 되는지
- **수정 방향**: 구체적 대안 제안 (가능하면 수정된 프롬프트/문장 예시 포함)
```

**톤/깊이 기준**: FEEDBACK.md에 연수님이 직접 작성한 피드백의 수준을 참조한다.
- 단순 "모호합니다"가 아니라, **왜** 모호한지, 학습자가 **어떤 경험**을 하게 되는지까지 서술
- 수정 방향은 **구체적 대안 텍스트** 수준으로 (가능한 경우)

## 드라이런 검증 체크리스트

드라이런 수행 시 각 모듈에서 아래 항목을 기계적으로 확인한다. 정적 분석의 "축 2: 실습 동작"을 **실제 실행으로 재검증**하는 자리.

| 체크 | 검증 도구 | 합격 기준 |
|------|----------|----------|
| 실습 프롬프트의 파일 경로가 실제 존재 | Read / Glob | 모든 경로가 200 OK로 읽힘 |
| 성공 기준의 숫자(파일 N개, 컬럼 N개)가 실제 값과 일치 | Read + 계수 | 숫자 일치 0건 불일치 |
| 검색·필터링 실습이 의미 있는 결과 반환 | Grep / 필터 실행 | 결과 ≥1건 |
| CSV 컬럼명이 프롬프트의 지시와 일치 | Read CSV 1행 | 컬럼 철자·순서 일치 |
| 프롬프트 코드블록 내부 줄 수 | awk 카운트 | 실습 프롬프트 ≤3줄 (설명용 예시 제외) |
| MCP 도구 가용성 (있는 경우) | ToolSearch | 스키마 확인 — 호출 없음 |
| MCP auth 플로우 단계 수 | 도구 설명 읽기 | 학습자 본문의 단계 수와 일치 |

**플래그 방식:** 체크 실패 시 `🟣 드라이런 발견` 배지로 기록. 심각도는 위 기준 그대로 (파일 없음 = 🔴, 길이 초과 = 🟡, 비유 보강 제안 = 🟢).
