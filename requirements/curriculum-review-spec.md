## Goal

cc-tutor 구조의 교육 커리큘럼을 **"Claude Code를 처음 쓰는 비개발자"** 관점에서 자동 리뷰하는 **범용** 재사용 스킬.

**핵심 질문: "이 워크숍을 듣고 나서, 학습자가 실제로 자기 업무에 적용해보려고 할까?"**

**설계 배경:**
- OM 교육에서 반이 "배운 건 좋았는데 실무 적용이 안 됨" — 기존 습관이 강해서 새 도구로 전환 안 됨
- 기능은 배우지만 "이걸 언제 어떻게 쓰지?"를 모르면 생각조차 안 남 (트리거 부재)
- cc-tutor-csm 전용이 아니라, cc-tutor 구조를 따르는 어떤 커리큘럼이든 리뷰 가능

---

## 리뷰 3축

| 축 | 체크 내용 |
|---|---|
| **초보자 눈높이** | 미설명 용어, 모호한 지시, 복붙 드릴 여부 |
| **실습 동작** | 참조 파일 존재, CSV 컬럼명 일치, 성공 기준 달성 가능 |
| **학습 흐름** | 모듈 간 연결, CLAUDE.md 규칙 활용, 점진적 복잡도 |

---

## 3대 안티패턴 (리뷰가 반드시 잡아야 할 것)

### 1. 복붙 드릴
학습자가 질문을 설계하지 않고, 주어진 프롬프트만 복사-붙여넣기하는 구조.

**판별 기준:**
- 실습 프롬프트가 "아래를 그대로 입력하세요"로만 구성되어 있는가?
- 학습자가 **본인 질문을 설계**할 기회가 있는가? (예: "궁금한 걸 골라서 물어보세요")
- 학습자가 **결과를 예측**해볼 기회가 있는가? (예: "이게 나올 것 같나요?")

**모듈 위치에 따른 판단 기준 (점진적 자유도):**
- M1(초반): 복붙 비중 높아도 OK — 도구에 익숙해지는 단계
- M2(중반): 복붙 50% + 자유 질문 50% — 도구 감각이 생긴 단계
- M3~M4(후반): 자유 질문/응용 비중이 더 높아야 함 — 복붙만이면 🔴

### 2. 실무-실습 갭
샘플 데이터로만 실습하고, 본인 업무 맥락으로 변환할 기회가 없는 구조.

**판별 기준:**
- `samples/` 고정 경로로만 실습하는가?
- "내 파일로 해봐도 됨" 유도 문구가 있는가?
- 실습 결과가 학습자의 **실제 산출물**(리포트, 메일 등)과 연결되는가?

### 3. 트리거 부재
기능은 가르치지만, "실무에서 언제 이걸 떠올려야 하는지" 연결이 없는 구조.

**판별 기준 — "실무 상황 + 도구"가 한 문장에 연결되어 있는가?**

| 트리거 있음 ✅ | 트리거 없음 ❌ |
|---|---|
| "매월 리포트 만들 때 이 스킬을 쓰세요" | "Write 도구로 파일을 만들 수 있습니다" |
| "파트장님 메시지가 오면 이 스킬을 실행" | "SKILL.md는 이런 구조입니다" |
| "기업 종료일이 다가오면 이걸로 확인" | "Grep은 파일 내용을 검색하는 도구입니다" |

**체크 위치:**
- 각 모듈의 WHY 섹션: "이 기능을 왜 배우는지" 실무 상황이 명시적인가?
- LEADER-TIP 섹션: "팀에서 언제 쓰면 되는지" 트리거가 있는가?
- NEXT-STEP 섹션: 다음 모듈로 넘어가는 연결이 실무 흐름인가?

## 좋은 실습의 4대 기준

| 기준 | 체크 | 예시 |
|------|------|------|
| **질문 설계** | 학습자가 스스로 질문을 만드는 구조인가 | "본인이 궁금한 걸 골라서 물어보세요" |
| **맥락 변환** | 본인 데이터/상황으로 바꿀 여지가 있는가 | "샘플 대신 my-data/에 넣어서 해봐도 됨" |
| **실무 트리거** | "이 상황이 오면 이걸 써"가 명시적인가 | "매월 리포트 만들 때 이걸 쓰면 돼" |
| **Before/After** | 시간 절약이 눈에 보이는가 | "수작업 6단계 → 한 마디" |

## 심각도 분류 체계

리뷰 결과의 각 항목에 심각도를 부여:

| 등급 | 의미 | 판단 기준 | 예시 |
|------|------|----------|------|
| **🔴 치명적** | 안 고치면 학습자가 이탈 | 실습이 물리적으로 안 돌아감, 완전한 복붙 드릴(후반 모듈), 설명 없이 전문용어 사용 | 참조 파일 없음, M4에서 복붙만, "MCP"를 설명 없이 사용 |
| **🟡 개선 필요** | 안 고쳐도 되지만 교육 효과 떨어짐 | 트리거 부재, 맥락 변환 기회 없음, 모듈 간 연결 약함 | WHY에 실무 상황 없음, samples/ 고정, M1 내용을 M2에서 안 씀 |
| **🟢 제안** | 더 좋아질 수 있는 포인트 | Before/After 강화 가능, 표현 개선, 추가 예시 가능 | 시간 절약 수치 추가, 비유 보강, 성공 기준 구체화 |

## 동기 부여 판단 기준 3가지

1. **Before/After 체감** — "수작업 1시간 → 2분" 같은 시간 절약이 눈에 보이는가
2. **내 업무 연결성** — 실습이 "내가 실제로 하는 일"과 직결되는가
3. **점진적 복잡도** — 쉬운 것→어려운 것 순서가 자연스러운가 (갑작스러운 점프 없는가)

## 산출물

- **파일**: `REVIEW.md` (별도 파일, FEEDBACK.md와 분리)
- **구조**: 모듈별 섹션 → 각 항목에 심각도 태그 + 3단계 포맷
  ```
  ## M2 데이터 분석

  ### 🔴 [실습 5] CLAUDE.md 취지에 역행하는 프롬프트
  - **문제**: 프롬프트에 CLAUDE.md 규칙을 다시 나열함
  - **왜 문제인가**: M1에서 "반복 설명 안 해도 됨"을 배웠는데, 실습이 그걸 무시
  - **수정 방향**: 프롬프트를 한 줄로 줄이고, CLAUDE.md가 자동 반영되는 걸 체감시키기
  ```
- **요약 섹션**: 리뷰 끝에 심각도별 집계 + 가장 먼저 고칠 것 Top 3
- **중복 처리**: FEEDBACK.md를 먼저 읽고, 이미 지적된 항목은 `⏭️ FEEDBACK.md에서 기지적` 태그 후 건너뜀

## 실행 방식

- `/커리큘럼-리뷰` → 전체 모듈 리뷰
- `/커리큘럼-리뷰 M2` → 특정 모듈만 리뷰
- 구조: cc-tutor 구조 고정 (references/module-*.md + samples/ + CLAUDE.md)

---

## 스킬이 읽는 파일

| 파일 | 용도 |
|------|------|
| `.claude/skills/*/references/module-*.md` | 리뷰 대상 (모듈 교안) |
| `.claude/skills/*/references/glossary.md` | 용어가 적절히 활용되는지 |
| `.claude/skills/*/SKILL.md` | 튜터 엔진 규칙과 모듈 내용 일관성 |
| `CLAUDE.md` | 규칙이 실습에서 실제 활용되는지 교차 체크 |
| `samples/*` | 실습에서 참조하는 파일이 존재하는지, 컬럼명 일치하는지 |
| `FEEDBACK.md` | 이미 지적된 항목 중복 제외용 |

## Decisions Made

| 질문 | 결정 |
|------|------|
| 리뷰 초점 | 초보자 눈높이 + 실습 동작 + 학습 흐름 (3개 모두) |
| 페르소나 깊이 | 흥미/동기 판단까지 (단순 용어 체크 아님) |
| 핵심 실패 시나리오 | 복붙만 하다 끝남 + 돌아가면 안 쓰게 됨 |
| 범용성 | cc-tutor 구조 고정, CSM 전용 아님 |
| 산출물 위치 | REVIEW.md (별도 파일) |
| 제안 수준 | 문제 + 이유 + 수정안 |
| 심각도 분류 | 🔴 치명적 / 🟡 개선 필요 / 🟢 제안 |
| 중복 처리 | FEEDBACK.md 참조하여 중복 제외 |
| 실행 단위 | 전체 또는 모듈 지정 |
| 복붙 판단 | 모듈 위치에 따라 기준 다름 (점진적 자유도) |
| 트리거 판별 | "실무 상황 + 도구"가 한 문장에 연결되어 있는가 |
